# Project — Machine in the Loop

### 키보드는 당신이 잡습니다. 기계가 루프를 돕습니다.

**jarvis**는 사람이 직접 코드를 짜는 동안 곁에서 변경을 지켜보다가,
의미 있는 순간에만 페어 내비게이터를 자동으로 깨우는 **Claude Code 워치 루프**입니다.
코드는 사람이 쓰고, AI는 관찰자로 머뭅니다.

🔗 **[웹에서 보기](https://pinguding.github.io/Project-Machine-in-the-loop/)**

`/loop /jarvis`  ·  `strength=medium`  ·  `risk=**/*payment*`  ·  `debounce=90s`

</div>

> **이것은 하루짜리 작업을 10분으로 줄이는 마법이 아닙니다.**
> 그러나 8시간짜리 작업을 3~4시간으로 — 그리고 그 속도를 *지속가능하게* — 만들 수는 있습니다.

## 철학 · The Philosophy

저자성(authorship)을 보존하는 AI 페어 개발 방식입니다. jarvis는 단순한 자동화 도구가 아니라,
하나의 입장에서 출발합니다 — **개발의 저자는 끝까지 사람이어야 합니다.**

이것은 AI Agent가 없던 과거로 돌아가자는 것이 아닙니다. **AX(AI Experience)를 사람 중심으로 재정의**하자는 것입니다.
시계를 되돌리는 것이 아니라 방향을 트는 것입니다.

### 왜 '자비스'인가 · The Name

자비스는 토니 스타크의 능력을 **증폭**시킵니다. 하지만 슈트를 입고 직접 **전투를 수행하고 판단하는 주체는
끝까지 토니**입니다 — 자비스에게서 정보와 보조를 받을 뿐, 조종간을 넘기지 않습니다. 자비스는 옆에서
띄워주고, 결정과 행동은 토니가 합니다.

지금의 Coding Agent는 그 반대로 가기 쉽습니다. 자동화 수준이 높아서가 아니라,
**"내 코드인데 내가 조종석에 없는"** 구조 — 사람을 의사결정 루프 *밖으로* 밀어내는 구조 — 를 만들기 때문입니다.

> 그래서 이름이 자비스입니다 — **사람을 조종석에 앉혀두는 AI.**

### Human in the loop ✗ vs Machine in the loop ✓

|  | **✗ Human in the loop** | **✓ Machine in the loop** |
|---|---|---|
| 구조 | AI가 짜고, 사람이 검토·승인합니다 | 사람이 짜고, 기계가 루프 안에서 보조합니다 |
| 결과 | 관찰자로서의 리뷰는 **허위 메타인지**를 만듭니다. 이해가 사람이 아니라 도구 안에 쌓이고, 오류는 이미 커밋된 코드가 되어 누군가 해체해야 합니다. | 구현이 사람 손에서 **장기 기억**으로 남습니다. 기계는 관찰자로 머물며 길을 안내하되, 키보드는 끝까지 사람이 잡습니다. |

핵심은 **루프의 소유권을 뒤집는 것**입니다. AI가 만든 루프 안에서 사람이 개입해 "리뷰"하는 것은
사람의 인지 구조상 가장 지기 쉬운 게임입니다 — 그래서 사람이 루프를 만들고, 기계가 그 안에서 보조합니다.

### 진단 — full automation의 실패 경로

성능이 기대만큼 나오지 않자 업계의 처방은 *"정제된 context를 주입해라 → prompt를 잘 다뤄라 →
context, context, 더 많은 context"* 로 갔고, AI-native 회사들은 거기서 **"사람은 코드를 짜지 않고
세세한 context만 준다"** 는 구조로 굳어졌습니다. 그 구조가 만드는 병은 이렇습니다.

- 버그가 나도 개발자가 **어디를 고칠지 인지하지 못합니다** → AI한테 곧장 "이 문제 해결해줘"로 달려갑니다.
- 단순 에러는 티가 나지 않지만, **다층적·복합적 원인**일 때 AI도 잡지 못하고 사람은 더 잡지 못합니다 → 해결 시간이 폭증합니다.
- Feature 8개를 뚝딱 만들 수는 있어도 **출시 가능한 최소 퀄리티에는 도달하지 못합니다.** 생성 속도가 곧 처리량은 아닙니다 — 병목이 "이해·디버깅"으로 옮겨갔는데 그 근육이 비어 있습니다.
- 결국 사람이 **주도권을 잃습니다.** 코드를 모르고, 짜지 않으니 "감"도 떨어지고, **주니어는 성장 기회를 박탈당합니다.**

문제의 경계선은 *"AI를 쓰느냐 마느냐"* 가 아닙니다. **"무엇을 위임할지 분류·판단하는 명령권을 사람이 쥐느냐"** 입니다.
보일러플레이트를 AI한테 생성시키는 것은 문제가 되지 않습니다. 그러나 그 *판단마저* "보편 Rule"로 자동화하려는 시도에는
역설적 모순이 너무 많습니다 — 위임 여부를 결정하는 주도권 자체를 다시 기계에 넘기는 것이기 때문입니다.
사람이 끝까지 **라우터/분류자**로 남아야 합니다.

### gray zone — 허위 메타인지의 다른 이름

**gray zone = 실제로는 모르는데, (스스로) 안다고 판정되는 영역입니다.**

- **생성(generation)** 은 "구현 위치 + 알고리즘 과정"을 **에피소드 기억**으로 남깁니다 — 길고 구체적입니다.
- **리뷰(reading)** 는 **재인(recognition) 수준의 친숙함**만 남기는데, 이것이 "이해"로 위장합니다.
  "훑어봤으니 안다"는 착각, 심지어 **"내가 만든 것 같다"는 저자성 착각**까지 생깁니다.

full automation이 누적될수록 이 gray zone가 넓어집니다. 그래서 사람의 인지를 다시 끌어올려야
지속가능성이 생깁니다 — jarvis의 가장 강한 축 중 하나입니다.

### 지속가능성 — "속도"가 아니라 "적분된 속도"

```
속도
 │      ╱‾‾╲          ← full automation: 초반 폭발(하루→10분)
 │     ╱    ╲___        그러나 gray zone가 넓어지며 속도 붕괴
 │    ╱         ╲__     ("8개 만들었지만 출시 퀄리티 미달"의 벽)
 │   ╱
 │  ╱━━━━━━━━━━━━━━━  ← Machine in the loop: 낮은 피크(8h→4h)지만 평평하게 유지
 │ ╱                    gray zone가 갇혀 있어 무너지지 않음
 └────────────────────▶ 시간
```

풀오토메이션의 속도는 **복리 이자가 붙은 대출**입니다. 지금 빠른 만큼 gray zone가 넓어지고,
유지보수·재작업("처음부터 다시")으로 이자까지 쳐서 되갚습니다. Machine in the loop은 피크는 낮아도
**갚을 빚을 만들지 않습니다.** *순간 속도*가 아니라 *적분된 속도(누적 처리량)*로 보면 이쪽이 이깁니다.

### 오류 비용의 비대칭 — 사람이 필터 자리에 있는가

같은 **3% 할루시네이션**이 두 구조에서 정반대로 작용합니다.

- **Full automation에서 3%:** 할루시네이션이 최종 산출물에 그대로 봉인됩니다. 사람이 짜지 않았으니(허위 메타인지) 걸러낼 위치에 있지도 않고, 그 3%가 커밋된 코드가 되어 누군가 발견하고 해체해야 합니다. 자동화 사이클마다 복리로 방향을 틉니다. → **치명적입니다.**
- **Navigator(jarvis)에서 3%:** 같은 3%가 **97%의 정확한 navigation 위에 얹힌 노이즈**일 뿐입니다. 여기서는 **사람이 필터 자리에 앉아 있습니다.** AI가 헛소리하면 **깔끔히 무시하고 할 일을 하면 끝입니다.** 폐기 비용은 ≈ 0이고, 산출물에는 사람이 직접 짠 코드만 남습니다. → **무해하고, 오히려 순이득입니다.**

두 구조를 가르는 진짜 축은 **"사람이 필터 자리에 있는가"** 입니다.

> **확인사살 — "모델이 좋아지면 다 해결된다"는 반론에 대해:**
> 더 좋은 AI는 gray zone를 줄이는 것이 아니라 **넓힙니다.** 코드가 더 그럴듯하고 정확해 보일수록
> "나 이거 안다"는 착각은 *더 설득력 있게* 걸려듭니다. 허접한 코드는 의심이라도 사지만, 완벽해 보이는
> 코드는 검증 욕구 자체를 꺼버립니다. 그래서 gray zone는 **모델 발전에 면역**입니다 — 이 논증의 무게는
> 한시적인 "3% 할루시네이션"이 아니라 영구적인 **인지 구조**에 실려 있습니다.

### 책임은 위임할 수 없습니다

그렇다면 사람이 wake 사이에 AI한테 다 짜게 시키고 리뷰만 받으면 어떻게 될까요? — 그것마저 **사람이 알아서 합니다.**
jarvis는 강제하지 않습니다. 강제하는 순간 그 자체가 사람에게서 선택권을 빼앗는 일이기 때문입니다.

대신 진짜 게이트는 도구의 이빨이 아니라 **책임**입니다.

> 노동(labor)은 위임할 수 있습니다. 그러나 **책임(accountability)은 위임할 수 없습니다.**
> 커밋에 내 이름이 박히는 순간, 그것은 내가 답해야 할 것이 됩니다.

사람은 **자기 이름 달고 나가는 것에 모르는 것을 답하지 않습니다.** 책임이 위임 불가능해지는 순간, 사람은
스스로 키보드 앞으로 끌려옵니다. 이것이 jarvis의 빠진 이빨을 대신하는 *구조적* 중력입니다.

이것을 "개인 규율에 떠넘기는 무책임한 설계"라 할 수도 있지만, 사실은 **정반대**입니다. 책임을 *분산*시키는 것은
풀오토메이션("AI가 짠 건데요")이고, Machine in the loop은 책임을 **한 사람에게 응축**시킵니다("네 이름, 네 커밋").
사람에게 책임을 *덜* 지우는 것이 아니라 *더* 지우는 구조입니다.

> **전제(정직하게):** 이 모든 것은 *커밋에 박힌 이름이 실제로 의미를 갖는 문화*(진짜 코드 리뷰,
> 자기 버그는 자기가 고침, on-call)를 전제합니다. 책임이 분산된 조직에서는 어떤 도구도 대신할 수 없습니다 —
> 다만 그것은 이미 피드백 루프가 끊긴 별개의 문제입니다.

### 거울 — 책임이 gray zone에 눈멀지 않게

책임이 사람을 끌어당기려면, 사람이 **자기가 모른다는 것을 알아야** 합니다. 그런데 gray zone의 본질이
바로 "모르는데 안다고 느낀다"입니다. 그래서 책임만으로는 부족할 수 있습니다.

거울은 그 틈을 메웁니다 — **강제가 아니라 책임을 정직하게 만드는 빛**으로 말입니다. 누적 변경의 상당 부분이
*생성된 코드*면, jarvis는 막지도 고치지도 않고 한 줄만 비춥니다.

```
🪞 이번 변경의 상당 부분이 이 세션에서 생성된 코드로 보여 — 네 이름으로 커밋될 텐데 직접 파악했는지 확인 필요.
```

키보드를 빼앗지 않으면서, 서명하는 사람이 **눈 뜨고 책임지게** 합니다. (`mirror=off`로 끌 수 있습니다 — 거울조차 비강제입니다.)

### 생산자가 아니라 안내자

Jarvis가 사람에게 주는 것은 셋뿐입니다. 공통점은 — 모두 **안내**이지 **생산**이 아니라는 것입니다.

| 놓친 것 짚기 | 더 나은 접근 제안 | 다음 스텝 안내 |
|---|---|---|
| "이 부분이 빠진 것 같아" | "이 로직이 더 효율적일 거야" | "다음엔 이걸 하면 될 것 같아" |

**페어코딩의 역할 그대로입니다.** 드라이버(사람)는 키보드를 잡고 직접 코드를 칩니다 — 영원히 사람의 자리입니다.
내비게이터(Jarvis)는 검토하고, 앞을 내다보고, 방향을 제안합니다. 규칙은 단 하나입니다.
**"내비게이터는 키보드를 빼앗지 않는다."**

### navigator의 어조 — 짚고 비킵니다

내비게이터는 **건조한 메모**로 짚습니다. 떠먹이지도(정답·지시), 떠보지도(퀴즈·질문) 않습니다 —
둘 다 사람을 수동적 응답자 자리에 앉히기 때문입니다. 염려되는 지점에 "확인 필요" 정도로 표시하고 비킵니다.
모르면 사람이 찾습니다(그 수단이 AI든 아니든 사람이 정합니다).

- ❌ 떠먹이기: *"40번째 줄에 null 체크 빠졌어, 이렇게 고쳐"*
- ❌ 떠보기: *"입력이 비어 있을 때 어떻게 될지 생각해봤어?"*
- ✅ 메모: *"입력이 비어 있을 수 있는데, 이 경우 확인 필요"*

> **책상 앞에 앉는 것을 도와주지, 책을 읽어주지는 않습니다.**

### 리트머스 테스트

기능을 더할 때마다 이렇게 묻습니다. ① 이것은 사람을 조종석에 더 앉히는가, 아니면 조종석에서 밀어내는가?
② 토니가 가라고 했는가(상시 허용 · 위임 · 금지 중 어디인가)? 통과하지 못하면, 아무리 편리해도 Jarvis가 아닙니다.

## 동작 원리 — 토큰을 아끼며 보다가, 새 변경·위험·커밋 때만 깨웁니다

폴링 자체는 `git diff` 한 번이라 토큰을 거의 쓰지 않습니다. 비싼 Jarvis 리뷰는 직전 리뷰 이후 새 변경이 쌓였거나·위험 경로가 바뀌었거나·커밋 경계를
넘었을 때만 돌려 **토큰 소비를 최소화**합니다.

1. **측정** — 매 wake마다 working tree 변경 라인 수, 위험 경로 변경, 현재 `HEAD`를 잽니다. 추적 변경 + **미추적(새) 파일**까지 합산합니다. 토큰은 거의 0입니다.
2. **경계 판단** — 새 커밋이 생겼는가? 변경을 되돌렸는가? 마커(`.jarvis/baseline`)와 비교해 자동 보정합니다.
3. **게이트 + debounce** — 새 변경·위험·커밋 경계 중 하나라도 충족하면 리뷰 대상입니다. 단 **대화 중이면 보류**하고, 멎으면 누적분을 한 번에 flush합니다.
4. **컨벤션·개인화 수집 → Jarvis** — 변경 파일의 AGENTS.md·README·경로별 rules + persona + 집중 영역을 모아 맥락으로 넘기고 Jarvis를 호출합니다. 결과는 마커로 감쌉니다.
5. **다음 wake 예약** — `ScheduleWakeup`으로 스스로 다음 실행 간격을 정합니다. `/loop /jarvis`로 띄우면 루프가 이어집니다.

> 반복은 **`/loop`의 self-paced 모드** 위에서 돌고, `ScheduleWakeup`이 매 간격을 정합니다. `ScheduleWakeup`은 `/loop`
> 안에서만 발화하므로 진입점은 항상 **`/loop /jarvis`** 입니다. 멈춤은 곧 `/loop` 인터럽트(Esc) — 다음 wake가 예약되지 않습니다.
> 다시 `/loop /jarvis`로 띄우면 재개되고, `/jarvis-reset`이 상태를 비웁니다.

## 핵심 기능 — 양이 아니라 위험에 반응합니다

| | 기능 | 설명 |
|---|---|---|
| 📏 | **새 변경 게이트** | 직전 리뷰 이후 **새 변경이 쌓이면**(분량 무관 — 고정 라인/파일 임계값 없음) 리뷰합니다. 한 번 리뷰한 변경은 다시 발동하지 않고, 새로 쌓일 때만 한 번씩 봅니다. 사람이 직접 무언가 쓴 뒤에야 AI가 등장합니다. |
| 🎯 | **위험 경로 트리거** | 결제·로그인 같은 **위험 경로**는 분량과 무관하게 단 한 줄만 바뀌어도 즉시 경고합니다. "작지만 치명적"을 놓치지 않습니다. |
| 🔒 | **커밋 경계 강제 관찰** | 새 커밋이 감지되면 변경 누적과 무관하게 1회 점검합니다. **착오가 코드로 봉인되기 직전**이 경고의 골든타임입니다. |
| ⏸️ | **대화 debounce** | 실시간 대화 중에는 리뷰를 미뤘다가, 멎으면 누적분을 한 번에 처리합니다. **흐름을 끊지 않습니다.** |
| 📚 | **컨벤션 자동 수집** | 변경 파일의 가장 가까운 AGENTS.md·디렉토리 README·경로별 팀 rules를 모아 주입합니다. "이 패키지 규칙"까지 알고 리뷰합니다. |
| ♾️ | **self-paced 워치** | `/loop /jarvis`로 띄우면 `ScheduleWakeup`이 다음 간격을 스스로 정해 **추가 입력 없이 계속 감시합니다.** |
| 🎭 | **내비게이터 개인화** | 빈 `persona.md`를 채워 Jarvis가 **누구처럼, 무엇에 예민하게** 볼지 정합니다. |
| 🔬 | **집중 영역** | `.claude/jarvis/focus/`에 "이 프로젝트에서 특히 봐야 할 것"을 모아두면 일반 카탈로그보다 **우선 렌즈**로 봅니다. |
| 🪞 | **gray zone 거울** | 누적 변경의 상당 부분이 **생성된 코드**면 "직접 파악했는지" 한 줄로 비춥니다. 막지도 고치지도 않고 거울만 비춥니다. |

## 투명성 — Jarvis 출력 마커

AI가 자동 생성한 리뷰는 일반 대화와 시각적으로 구분됩니다. 마커 사이가 Jarvis의 말이고,
바깥은 워치의 운영 메시지입니다. 끝 문구가 매번 저자성을 상기시킵니다.

```
# 대화가 멎자 보류했던 리뷰가 flush 됩니다
╭─🤖 JARVIS ─────────────── (jarvis 자동 관찰)
입력이 비어 있을 수 있는데 checkout.total()가 빈 배열을 그대로
참조해 크래시로 이어질 수 있어 보여 — 빈 입력일 때 이 경로 확인 필요.
· 참고: packages/checkout/AGENTS.md (경계 케이스 규칙)
╰─🤖 JARVIS ─────────────── 키보드는 당신이 잡습니다 · 판단은 당신 몫
```

**관찰자로 머뭅니다.** Jarvis는 코드를 직접 고치지 않습니다. 검토·지적·다음 스텝 제안만 하고,
"이거 고쳐줘"라고 **사람이 위임할 때만** 구현으로 전환합니다.

---

## 구성 — `.claude/skills/`

| 스킬 | 역할 |
|------|------|
| **jarvis** | 진입점 워치 루프입니다. `/loop /jarvis`로 띄웁니다. `git diff`로 싸게 폴링하다 새 변경 누적/위험경로/커밋경계를 넘으면 `jarvis-once`를 호출하고, `ScheduleWakeup`으로 다음 tick 간격을 정합니다. |
| **jarvis-once** | 단발 페어 내비게이터입니다. 검토·지적·다음 스텝 제안만 하고 **코드를 직접 쓰지 않습니다.** |
| **jarvis-reset** | `.jarvis/` 상태(baseline·args)를 삭제해 완전 초기화하는 독립 명령입니다. 돌고 있는 루프를 멈추진 않으며(그건 Esc), 멈춘 뒤 깨끗하게 새로 시작할 때 씁니다. |

상태는 워킹 트리의 `.jarvis/`(gitignored, 클론마다 독립적인 로컬 상태)에 저장됩니다.
`.git/`을 쓰지 않는 이유는, `.git/` 내부 쓰기가 매번 권한 승인을 요구해 무인 루프에 부적합하기 때문입니다.

## 개인화 — 두루뭉술한 보편 맥락을 믿지 않습니다

정확한 navigation은 구체적이고 고유한 맥락에서 나옵니다. 그래서 jarvis는 하드코딩된 보편 규칙 대신
**프로젝트가 실제로 가진 문서**(CLAUDE.md·AGENTS.md·경로별 rules 등)를 읽고, 두 레이어로 개인화합니다.

- **`.claude/skills/jarvis-once/persona.md`** — 내비게이터가 *누구인지* + *무엇에 예민한지*를 정합니다.
  **빈 채로 제공**되며, 채우면 그 사람/팀에 맞게 리뷰 성격이 바뀝니다. (비면 기본 내비게이터로 동작합니다.)
- **`.claude/jarvis/focus/`** — *이 프로젝트에서* 특히 주의 깊게 봐야 할 내용을 모아두는 디렉토리입니다.
  일반 감지 카탈로그보다 **우선**해서 봅니다. 커밋되어 팀과 공유할 수 있습니다. `/loop /jarvis focus=<경로>`로 위치를 바꿉니다.

무엇을 개인화하든 **"코드 생산 금지 · 떠먹이지도 떠보지도 않는다"** 는 핵심 규칙은 유지됩니다.

---

## 설정 — args 한 줄로 튜닝

```
/loop /jarvis strength=high risk=**/*payment* debounce=90s
```

| 키 | 의미 | 기본값 |
|---|---|---|
| `strength` | 강도 프리셋 — 아래 폴링 간격 노브들을 묶어 세팅합니다 (`low`/`medium`/`high`) | `medium` |
| `risk` | 위험 경로 glob — 분량 무관 즉시 경고 | `off` |
| `debounce` | 대화가 멎었다고 볼 정지 시간 | `90s` |
| `active` | 워킹트리에 변경 있음 / 리뷰 직후 wake 간격 | `4m` |
| `idle` | 워킹트리 깨끗할 때 wake 간격 | `25m` |
| `warmup` | 첫 로드 빠른 폴링 tick 수 | `3` |
| `paths` | 감시 대상 경로 한정 | 전체 |
| `focus` | 집중 영역 디렉토리 | `.claude/jarvis/focus/` |
| `mirror` | gray zone 거울 (비강제) | `on` |

> 💡 폴링을 **촘촘히 해도 잔소리는 늘지 않습니다.** Jarvis는 자체 severity 문턱 아래선 스스로 침묵하기 때문에,
> 촘촘한 폴링은 *호출 비용*만 늘릴 뿐 노이즈를 만들지 않습니다. 반응 속도를 높이고 싶으면 안심하고 강도를 올리세요.

## 사용

```
/loop /jarvis                # 최초 1회 강도(strength)를 물어본 뒤 워치 시작
/loop /jarvis strength=high   # 작은 변화도 자주 점검
/loop /jarvis strength=low    # 큰 덩어리만 드물게
/loop /jarvis risk=**/*payment*  # 위험 경로는 분량 무관 즉시 경고
/loop /jarvis mirror=off      # gray zone 거울 끄기 (기본 on)

# 멈춤·일시정지  → 루프 인터럽트(Esc)
# 재개          → /loop /jarvis 다시 (저장된 설정 복원)
# 완전 초기화    → /jarvis-reset
```

> ⚠️ **`/loop`에 간격을 직접 주지 마세요.** `/loop /jarvis`처럼 **간격 없이** 띄워야 self-pace로 돌아 jarvis의 `active`/`idle`/`strength` args가 `ScheduleWakeup`을 통해 주기를 몹니다. `/loop 5m /jarvis`는 고정 스케줄로 돌아 그 args를 **무시**합니다. 주기 조절은 평소처럼: `/loop /jarvis active=3m idle=20m`.

### 설치

설치 시 **언어를 선택**합니다(영문 또는 한글 스킬). 선택하면 현재 프로젝트(`.claude/skills/`)에 설치하고,
집중 영역 디렉토리를 만들고, `.jarvis/`를 gitignore에 추가합니다.

한 줄이면 됩니다(이 레포를 자기복제):

```bash
curl -fsSL https://raw.githubusercontent.com/pinguding/Project-Machine-in-the-loop/main/install.sh | bash
```

또는 clone 후 옵션과 함께 실행합니다.

```bash
git clone https://github.com/pinguding/Project-Machine-in-the-loop.git
./Project-Machine-in-the-loop/install.sh                  # 현재 프로젝트, 언어를 물어봄
./Project-Machine-in-the-loop/install.sh --lang ko        # 한글 스킬, 프롬프트 없이
./Project-Machine-in-the-loop/install.sh --lang en <path> # 영문 스킬을 지정 프로젝트에
./Project-Machine-in-the-loop/install.sh --global         # ~/.claude/skills/ 전역 설치 (모든 프로젝트)
```

그다음 그 프로젝트에서 Claude Code를 열고 `/loop /jarvis`를 실행하면 됩니다. (설치 스크립트) 재실행은 안전하며(멱등성), 스킬을 제자리에서 갱신합니다.

> 스킬 소스는 `.claude/`에 숨기지 않고 **오픈된 공간** `skills/en/` · `skills/ko/`에 둡니다. 설치 스크립트가
> 선택한 언어를 대상의 `.claude/skills/`로 복사하며, Claude Code는 거기서 스킬을 인식합니다.

## 닫는 질문 · The Closing Question

> **"AI 서비스가 멈춰 있는 몇 시간 동안, 우리의 작업도 멈춰 있어야 합니까?"**

full automation에 깊이 기댈수록 그 몇 시간 동안 우리는 **아무것도 하지 못합니다** — 이해가 사람이 아니라
도구 안에 쌓여 있기 때문입니다. 도구가 잠시 사라져도 흔들림 없이 손을 움직이려면, 코드에 대한 이해는
**도구 안보다 우리 안에** 더 많이 쌓여 있어야 하지 않을까요?

**jarvis는 그 질문에 대한 하나의 대답입니다.**
