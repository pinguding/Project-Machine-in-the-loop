Why We Built It · The Philosophy

Machine in the Loop

An approach to AI pair development that preserves authorship. jarvis isn't just an automation tool — it starts from a single stance: the author of the code must remain the human, all the way through.

Why 'Jarvis' · The Name

Jarvis amplifies Tony Stark's abilities. But the one who suits up, fights, and makes the calls is always Tony — he takes information and assistance from Jarvis, but he never hands over the controls. Jarvis lifts him up from the side; the decisions and the actions are Tony's.

Today's coding agents tend to go the opposite way. Not because automation is too capable, but because they build a "it's my code, yet I'm not in the cockpit" structure — one that pushes the human out of the decision loop.

That's why the name is Jarvis — an AI that keeps the human in the cockpit.

✗ Human in the loop

The AI writes; the human reviews and approves

Reviewing as an observer creates false metacognition. Understanding accumulates inside the tool rather than in the person, and errors become already-committed code that someone has to unwind.

✓ Machine in the loop

The human writes; the machine assists from inside the loop

The implementation lives on as long-term memory in the human's hands. The machine stays an observer, pointing the way — but the human holds the keyboard all the way through.

Background배경

As AI full-automation development spreads fast, the author of code generation is shifting from human to AI. The conventional "Human in the loop" is a structure where the AI produces code and the human reviews and approves it.

Problem문제

This structure works against the developer's grasp of the logic. As automation cycles pile up, the developer knows the overall context but loses awareness of the detailed logic.

More dangerous still, reviewing as an "observer" rather than an "author" creates the illusion of "knowing" — false metacognition — and this blind spot accelerates as scale and change grow.

A developer who has lost the context drifts toward ever more abstract instructions, and after rounds of ping-pong with the AI the result veers off, so the cost of starting over compounds.

Approach접근

We flip the roles to build Machine-in-the-loop instead of Human-in-the-loop. The author of the development is the human, and the machine assists from inside the loop.

The human writing the code directly is the founding principle, while the AI continuously observes the logic in progress and offers Review · Navigation · Warning — but it intervenes in the code only when the user asks. Otherwise it stays an observer.

Crucially, Navigation offers reference knowledge and patterns, not a finished answer — it shows without spoon-feeding, preserving authorship.

Mechanism작동 원리

Because the human writes it directly, the implementation's locations and the path through the algorithm remain as long, concrete long-term memory — the exact opposite of false metacognition.

The cost of errors is also asymmetric. An AI hallucination can simply be ignored by the author, but a full-automation error becomes already-committed code that someone has to find and unwind. When the human slips, the AI actively leaves a warning.

Impact기대 효과

The developer can focus more on the real work, and the anxiety over black-boxed logic disappears.

On speed, it won't turn a day's work into ten minutes — it turns an 8-hour job into 3–4 hours, but sustains that consistently. Even as scale grows and change accelerates, context forgetting and loss shrink, so we can always give the AI clear instructions.

And the developer can take responsibility for their own code.

A Guide, Not a Producer

Jarvis gives the human only three things. What they share is — they're all guidance, not production.

Flagging what's missing

"I think this part is left out"

Suggesting a better approach

"This logic would be more efficient"

Pointing to the next step

"Next, I think you could do this"

The pair-coding roles, exactly as they are. The driver (the human) holds the keyboard and types the code directly — forever the human's seat. The navigator (Jarvis) reviews, looks ahead, and suggests direction. There is only one rule: "The navigator never takes the keyboard."
The litmus test — with every feature we add, we ask: ① Does this seat the human deeper in the cockpit, or push them out of it? ② Did Tony say to go (is it always-allowed, delegated, or forbidden)? If it doesn't pass, then however convenient it is, it isn't Jarvis.
How It Works

Watch cheaply, wake only when it's risky

Polling itself is a single git diff, so it barely uses any tokens. The expensive Jarvis review runs only when a threshold, risk, or commit boundary is crossed, keeping token consumption to a minimum.

1

Measure

On every wake, gauge the working tree's changed lines and file count, risky-path changes, and the current HEAD. Almost zero tokens.

2

Judge boundaries

Was a new commit made? Were changes reverted? Compare against the marker (.git/jarvis-baseline) and auto-correct.

3

Gate + debounce

If any one of threshold, risk, or commit boundary is met, it's a review candidate. But if a conversation is underway, hold, then flush the accumulated changes all at once when it settles.

4

Gather conventions → Jarvis

Collect the changed files' AGENTS.md, README, and per-path rules, pass them as context, and call Jarvis. The result is wrapped in markers.

5

Schedule the next wake

It schedules its own next run with ScheduleWakeup. Self-repeating on its own, with no /loop wrapper.

Core Features

Reacts to risk, not volume

GATE📏

Threshold gate

When changes accumulated since the last review cross the line-count or file-count threshold, it reviews. The AI shows up only after the human has personally written a meaningful amount.

RISK🎯

Risky-path trigger

Risky paths like payments and login warn instantly the moment even one line changes, regardless of volume. It never misses the "small but fatal."

COMMIT🔒

Enforced commit-boundary observation

When a new commit is detected, it checks once even below threshold. The moment just before a slip gets sealed into code is the golden time for a warning.

DEBOUNCE⏸️

Conversation debounce

It defers reviews during a live conversation, then delivers the accumulated changes all at once when the conversation settles. It doesn't break your flow.

CONTEXT📚

Automatic convention gathering

It gathers the changed files' nearest AGENTS.md, directory README, and per-path team rules and injects them into Jarvis. It reviews knowing even "this package's rules."

LOOP♾️

Standalone self-repetition

ScheduleWakeup is the engine of the loop. Without prefixing /loop, one call and it runs on its own.

PERSONA🎭

Navigator personalization

Fill in the empty persona.md to set who Jarvis looks like, and what it's sensitive to. Navigation tuned to your own hands, not a vague, universal context.

FOCUS🔬

Focus areas

Collect "what especially deserves attention in this project" under .claude/jarvis/focus/, and it looks through that as a priority lens over the general catalog.

MIRROR🪞

Gray-zone mirror

When a large share of the accumulated changes is generated code, it reflects, in a single line, whether you've actually grasped it. It neither blocks nor fixes — just a mirror, so responsibility opens its eyes.

Transparency

The Jarvis output marker

An AI-generated review is visually distinct from ordinary conversation. What's between the markers is Jarvis speaking; what's outside is the watch's operational messaging. The closing line reminds you of authorship every time.

# Once the conversation settles, the held review flushes ╭─🤖 JARVIS ─────────────── (jarvis auto-observation) The input may be empty, and checkout.total() references the empty array as-is, which could lead to a crash — worth checking this path on empty input. · ref: packages/checkout/AGENTS.md (edge-case rules) ╰─🤖 JARVIS ─────────────── You hold the keyboard · the judgment is yours
It stays an observer. Jarvis doesn't fix the code itself. It only reviews, flags, and suggests the next step, and it switches to implementation only when the human delegates with "go fix this." It neither spoon-feeds the answer nor probes you with a quiz — it flags the concern with a dry memo. To preserve authorship.

The Closing Question

“During the few hours an AI service is down,
should our work be down too?”

The deeper we lean on full automation, the more those few hours leave us unable to do anything — because the understanding has accumulated inside the tool rather than in the person. For our hands to keep moving, unshaken, even when the tool briefly disappears, shouldn't our understanding of the code accumulate more inside us than inside the tool?

jarvis is one answer to that question.
