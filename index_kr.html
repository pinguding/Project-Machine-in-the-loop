jarvis는 사람이 직접 코드를 짜는 동안 곁에서 변경을 지켜보다가, 의미 있는 순간에만 페어 내비게이터 Jarvis를 자동으로 깨우는 워치 루프입니다. 코드는 사람이 쓰고, AI는 관찰자로 머뭅니다.

왜 만들었나 · The Philosophy

Machine in the Loop

저자성(authorship)을 보존하는 AI 페어 개발 방식. jarvis는 단순한 자동화 도구가 아니라, 하나의 입장에서 출발합니다 — 개발의 저자는 끝까지 사람이어야 한다.

왜 '자비스'인가 · The Name

자비스는 토니 스타크의 능력을 증폭시킵니다. 하지만 슈트를 입고 직접 전투를 수행하고 판단하는 주체는 끝까지 토니입니다 — 자비스에게서 정보와 보조를 받을 뿐, 조종간을 넘기지 않습니다. 자비스는 옆에서 띄워주고, 결정과 행동은 토니가 합니다.

지금의 Coding Agent는 그 반대로 가기 쉽습니다. 자동화 수준이 높아서가 아니라, "내 코드인데 내가 조종석에 없는" 구조 — 사람을 의사결정 루프 밖으로 밀어내는 구조 — 를 만들기 때문입니다.

그래서 이름이 자비스입니다 — 사람을 조종석에 앉혀두는 AI.

✗ Human in the loop

AI가 짜고, 사람이 검토·승인

관찰자로서의 리뷰는 허위 메타인지를 만듭니다. 이해가 사람이 아니라 도구 안에 쌓이고, 오류는 이미 커밋된 코드가 되어 누군가 해체해야 합니다.

✓ Machine in the loop

사람이 짜고, 기계가 루프 안에서 보조

구현은 사람 손에서 장기 기억으로 남습니다. 기계는 관찰자로 머물며 길을 안내하되, 키보드는 끝까지 사람이 잡습니다.

Background배경

AI full automation 개발이 빠르게 확산되며 코드 생성의 주체가 사람에서 AI로 이동하고 있습니다. 통상의 "Human in the loop"는 AI가 코드를 생산하고 사람이 이를 검토·승인하는 구조입니다.

Problem문제

이 구조는 개발자의 로직 인지능력과 반대 방향으로 작용합니다. 자동화 사이클이 누적될수록 개발자는 전체 맥락은 알지만 상세 로직에 대한 인지를 잃습니다.

더 위험한 것은, "저자"가 아닌 "관찰자"로서의 리뷰가 "알고 있다"는 착각 — 허위 메타인지(false metacognition) — 을 만든다는 점입니다. 이 맹점은 규모와 변경량이 커질수록 가속됩니다.

맥락을 잃은 개발자는 점점 추상적인 지시만 내리게 되고, AI와의 핑퐁 끝에 결과가 어긋나 처음부터 다시 시작하는 비용이 복리로 쌓입니다.

Approach접근

역할을 뒤집어 Human-in-the-loop이 아닌 Machine-in-the-loop을 만듭니다. 개발의 저자는 사람이고, 기계는 루프 안에서 보조합니다.

사람이 직접 개발하는 것을 기본 원칙으로 하며, AI는 개발 중인 로직을 지속적으로 관찰하며 Review · Navigation · Warning 을 제공하되, 코드에 직접 개입하는 것은 사용자의 요청이 있을 때뿐입니다. 그 외에는 관찰자로 머뭅니다.

결정적으로 Navigation은 정답 구현이 아니라 참고 지식·패턴을 제시합니다 — 보여주되 떠먹이지 않아 저자성을 보존합니다.

Mechanism작동 원리

사람이 직접 쓰기 때문에 구현 위치와 알고리즘 과정이 길고 구체적인 장기 기억으로 남습니다 — 허위 메타인지의 정반대입니다.

오류 비용도 비대칭입니다. AI의 할루시네이션은 저자가 무시하면 그만이지만, full automation의 오류는 이미 커밋된 코드가 되어 누군가 발견하고 해체해야 합니다. 사람의 착오에는 AI가 적극적으로 경고를 남깁니다.

Impact기대 효과

개발자는 본연의 업무에 더 집중할 수 있고, 블랙박스화된 로직에 대한 불안은 사라집니다.

속도는 하루짜리 작업을 10분으로 줄여주지는 않습니다 — 8시간 걸릴 일을 3~4시간으로, 그러나 꾸준히 유지합니다. 규모가 커지고 변화가 많아져도 컨텍스트 망각·유실이 줄어들어, 우리는 AI에게 항상 명확한 지시를 줄 수 있습니다.

그리고 개발자는 자신의 코드에 책임질 수 있습니다.

생산자가 아니라 안내자

Jarvis가 사람에게 주는 것은 셋뿐입니다. 공통점은 — 모두 안내이지 생산이 아니라는 것.

놓친 것 짚기

"이 부분이 빠진 것 같아"

더 나은 접근 제안

"이 로직이 더 효율적일 거야"

다음 스텝 안내

"다음엔 이걸 하면 될 것 같아"

페어코딩의 역할 그대로. 드라이버(사람)는 키보드를 잡고 직접 코드를 칩니다 — 영원히 사람의 자리. 내비게이터(Jarvis)는 검토하고, 앞을 내다보고, 방향을 제안합니다. 규칙은 단 하나: "내비게이터는 키보드를 빼앗지 않는다."
리트머스 테스트 — 기능을 더할 때마다 묻습니다. ① 이건 사람을 조종석에 더 앉히는가, 아니면 조종석에서 밀어내는가? ② 토니가 가라고 했나 (상시 허용·위임·금지 중 어디인가)? 통과 못 하면, 아무리 편리해도 Jarvis가 아닙니다.
동작 원리

토큰을 아끼며 보다가, 새 변경·위험·커밋 때만 깨운다

폴링 자체는 git diff 한 번이라 토큰을 거의 쓰지 않습니다. 비싼 Jarvis 리뷰는 직전 리뷰 이후 새 변경이 쌓였거나·위험 경로가 바뀌었거나·커밋 경계를 넘었을 때만 돌려 토큰 소비를 최소화합니다.

1

측정

매 wake마다 working tree 변경 라인 수, 위험 경로 변경, 현재 HEAD를 잰다. 토큰 거의 0.

2

경계 판단

새 커밋이 생겼나? 변경을 되돌렸나? 마커(.jarvis/baseline)와 비교해 자동 보정.

3

게이트 + debounce

새 변경·위험·커밋 경계 중 하나라도 충족하면 리뷰 대상. 단 대화 중이면 보류하고, 멎으면 누적분을 한 번에 flush.

4

컨벤션 수집 → Jarvis

변경 파일의 AGENTS.md·README·경로별 rules를 모아 맥락으로 넘기고 Jarvis를 호출. 결과는 마커로 감싼다.

5

다음 wake 예약

ScheduleWakeup으로 스스로 다음 실행을 건다. /loop 래퍼 없이 단독 자기 반복.

핵심 기능

양이 아니라 위험에 반응한다

GATE📏

새 변경 게이트

직전 리뷰 이후 새 변경이 쌓이면(분량 무관 — 고정 라인/파일 임계값 없음) 리뷰. 한 번 리뷰한 변경은 다시 발동하지 않고 새로 쌓일 때만 본다. 사람이 직접 무언가 쓴 뒤에야 AI가 등장.

RISK🎯

위험 경로 트리거

결제·로그인 같은 위험 경로는 분량과 무관하게 단 한 줄만 바뀌어도 즉시 경고. "작지만 치명적"을 놓치지 않는다.

COMMIT🔒

커밋 경계 강제 관찰

새 커밋이 감지되면 변경 누적과 무관하게 1회 점검. 착오가 코드로 봉인되기 직전이 경고의 골든타임.

DEBOUNCE⏸️

대화 debounce

실시간 대화 중에는 리뷰를 미뤘다가, 대화가 멎으면 그때 누적분을 한 번에. 흐름을 끊지 않는다.

CONTEXT📚

컨벤션 자동 수집

변경 파일의 가장 가까운 AGENTS.md·디렉토리 README·경로별 팀 rules를 모아 Jarvis에 주입. "이 패키지 규칙"까지 알고 리뷰.

LOOP♾️

단독 자기 반복

ScheduleWakeup이 반복의 엔진. /loop을 앞에 붙이지 않아도 한 번 호출로 스스로 돈다.

PERSONA🎭

내비게이터 개인화

빈 persona.md를 채워 Jarvis가 누구처럼, 무엇에 예민하게 볼지 정한다. 두루뭉술한 보편 맥락이 아니라 내 손에 맞춘 navigation.

FOCUS🔬

집중 영역

.claude/jarvis/focus/에 "이 프로젝트에서 특히 봐야 할 것"을 모아두면, 일반 카탈로그보다 우선 렌즈로 본다.

MIRROR🪞

gray zone 거울

누적 변경의 상당 부분이 생성된 코드면 "직접 파악했는지" 한 줄로 비춘다. 막지도 고치지도 않고 거울만 — 책임이 눈 뜨게.

투명성

Jarvis 출력 마커

AI가 자동 생성한 리뷰는 일반 대화와 시각적으로 구분됩니다. 마커 사이가 Jarvis의 말, 바깥은 워치의 운영 메시지. 끝 문구가 매번 저자성을 상기시킵니다.

# 대화가 멎자 보류했던 리뷰가 flush 됩니다 ╭─🤖 JARVIS ─────────────── (jarvis 자동 관찰) 입력이 비어 있을 수 있는데 checkout.total()가 빈 배열을 그대로 참조해 크래시로 이어질 수 있어 보여 — 빈 입력일 때 이 경로 확인 필요. · 참고: packages/checkout/AGENTS.md (경계 케이스 규칙) ╰─🤖 JARVIS ─────────────── 키보드는 당신이 잡습니다 · 판단은 당신 몫
관찰자로 머문다. Jarvis는 코드를 직접 고치지 않습니다. 검토·지적·다음 스텝 제안만 하고, "이거 고쳐줘"라고 사람이 위임할 때만 구현으로 전환합니다. 정답을 떠먹이지도, 퀴즈로 떠보지도 않고 — 염려되는 지점을 건조한 메모로 짚습니다. 저자성을 보존하기 위해.
설정

args 한 줄로 튜닝

/jarvis strength=high risk=**/*payment* debounce=90s

키의미기본값
strength강도 프리셋 — 아래 폴링 간격 노브들을 묶어 세팅 (low/medium/high)medium
risk위험 경로 glob — 분량 무관 즉시 경고off
debounce대화가 멎었다고 볼 정지 시간90s
active워킹트리에 변경 있음 / 리뷰 직후 wake 간격4m
idle워킹트리 깨끗할 때 wake 간격25m
warmup첫 로드 빠른 폴링 tick 수3

💡 폴링을 촘촘히 해도 잔소리는 늘지 않습니다. Jarvis는 자체 severity 문턱 아래선 스스로 침묵하기 때문에, 촘촘한 폴링은 호출 비용만 늘릴 뿐 노이즈를 만들지 않습니다. 반응 속도를 높이고 싶으면 안심하고 강도를 올리세요.
닫는 질문 · The Closing Question

“AI 서비스가 멈춰 있는 몇 시간 동안,
우리의 작업도 멈춰 있어야 합니까?”

full automation에 깊이 기댈수록 그 몇 시간 동안 우리는 아무것도 하지 못합니다 — 이해가 사람이 아니라 도구 안에 쌓여 있기 때문입니다. 도구가 잠시 사라져도 흔들림 없이 손을 움직이려면, 코드에 대한 이해는 도구 안보다 우리 안에 더 많이 쌓여 있어야 하지 않을까요?

jarvis는 그 질문에 대한 하나의 대답입니다.
